## 1. 先理解四个核心概念

### <Activity className="inline-block w-4 h-4 align-text-bottom text-primary" /> Workflow 驱动，而不是硬编码流程
总结、实体提取、RAG 检索都由工作流步骤组合执行，便于替换、扩展和调试。

### <Database className="inline-block w-4 h-4 align-text-bottom text-primary" /> 本地数据库是主存
记忆、实体等核心数据写入 IndexedDB；WorldBook 主要承担宏占位与接入桥梁角色。

### <Search className="inline-block w-4 h-4 align-text-bottom text-primary" /> 召回由 Retriever 统一调度
可走向量、混合或 Agentic 直通路径。召回后统一进入宏缓存，参与当前轮上下文。

### <WandSparkles className="inline-block w-4 h-4 align-text-bottom text-primary" /> 预处理是“发送前增强”
在 GENERATION_AFTER_COMMANDS 阶段拦截输入，提取 `<output>` / `<query>`，再继续生成。

## 2. 配置流程（按顺序）

### 第一步：模型接入（API Presets）
进入 **API 预设** 页面，至少完成：

1. **LLM 预设**（必需）
   - 供 Summary / Entity / Preprocessing 使用。
2. **Embedding**（强烈建议必配）
   - 没有它，RAG 召回能力会明显下降。
3. **Rerank**（可选）
   - 相关性不稳定时开启，提高结果排序质量。

### 第二步：运行时策略（Processing）
进入 **数据处理 → 运行时配置**：

- **自动总结**：建议开启。
- **楼层间隔**：推荐 `20 ~ 30`。
- **缓冲层数**：推荐 `10 ~ 15`（给你留修文空间）。
- **Trimmer**：建议开启，用于记忆库长期控量与层级压缩。

### 第三步：记忆流检查（Memory Stream）
进入 **记忆编辑**：

- 查看是否有新 `EventNode` / `EntityNode` 生成。
- 手动编辑后记得 **Re-embed**，否则检索向量仍是旧版本。

---

## 3. 预处理与 RAG 联动（推荐开启）

### <Play className="inline-block w-4 h-4 align-text-bottom text-primary" /> 何时开启
如果你需要“输入润色、指代展开、剧情编排”，建议开启预处理。

### <Sliders className="inline-block w-4 h-4 align-text-bottom text-primary" /> 最小配置
- 选择一个 `preprocessing` 分类模板。
- 保证模板输出含 `<output>` 标签。
- Query 增强场景补充 `<query>` 标签。

### <Settings className="inline-block w-4 h-4 align-text-bottom text-primary" /> 预期行为
- `<output>`：替换本轮用户输入。
- `<query>`：进入检索流程，提升召回命中率。

---

## 4. 3 条验证用例（建议立即执行）

1. 连续发送若干消息，确认达到间隔后出现总结任务。
2. 输入“我们之前约定的那件事”，检查召回日志是否返回相关历史。
3. 手动修改一条摘要并 Re-embed，再搜索新关键词验证检索更新。

---

## 5. 常见误区（新版）

- **误区 1**：只配 LLM 不配 Embedding。
  - 结果：能总结，但检索弱或几乎无效。
- **误区 2**：间隔设太小。
  - 结果：摘要碎片化严重，后续 Trimmer 压力变大。
- **误区 3**：编辑文本后不重嵌。
  - 结果：你看到的新文本与向量索引不一致，搜索“像坏了一样”。
