## 1. 先理解四个核心概念

### <Activity className="inline-block w-4 h-4 align-text-bottom text-primary" /> 工作流驱动，而不是硬编码流程
总结、实体提取、检索召回都由工作流步骤组合执行，便于替换、扩展和调试。

### <Database className="inline-block w-4 h-4 align-text-bottom text-primary" /> 本地数据库是主存
记忆、实体等核心数据写入浏览器的本地数据库；酒馆原生的世界书主要只承担宏变量占位与接入桥梁的角色。

### <Search className="inline-block w-4 h-4 align-text-bottom text-primary" /> 召回由统一门面服务调度
可走向量打分、混合重排或主动直通（Agentic）路径。召回结果统一进入宏变量缓存池，参与当前轮的上下文。

### <WandSparkles className="inline-block w-4 h-4 align-text-bottom text-primary" /> 预处理是“发送前增强”
在用户发言发出前的阶段拦截输入，大模型分析后提取 `<output>` 正文和 `<query>` 搜索词，再继续系统生成。

## 2. 配置流程（按顺序）

### 第一步：模型接入
进入 **API 预设** 页面，至少完成：

1. **大语言模型 (LLM) 预设**（必需）
   - 供总结 / 实体提取 / 意图预处理 使用。
2. **向量化 (Embedding) 模型**（强烈建议必配）
   - 没有它，文本特征关联检索的能力会明显下降。
3. **重排序 (Rerank) 模型**（可选）
   - 相关性不稳定时开启，大幅提高候选结果排序质量。

### 第二步：运行时策略
进入 **数据处理 → 运行时配置**：

- **自动总结**：建议开启。
- **楼层间隔**：推荐 `20 ~ 30`。
- **缓冲层数**：推荐 `10 ~ 15`（给你留出修文空间）。
- **合并与压缩机制 (Trimmer)**：建议开启，用于记忆库长期控量与层级架构的维护。

### 第三步：记忆流检查
进入 **记忆编辑**：

- 查看是否有新的事件节点或实体节点生成。
- 手动编辑后记得 **重新向量化 (Re-embed)**，否则底层的检索向量仍是旧版本。

---

## 3. 预处理与检索联动（推荐开启）

### <Play className="inline-block w-4 h-4 align-text-bottom text-primary" /> 何时开启
如果你需要“输入润色、含糊词（如：那个、他）指代展开、剧情前置编排”，建议开启预处理。

### <Sliders className="inline-block w-4 h-4 align-text-bottom text-primary" /> 最小配置
- 选择一个意图预处理分类的模板。
- 保证判定模型输出含 `<output>` 标签。
- 搜索增强场景补充 `<query>` 标签。

### <Settings className="inline-block w-4 h-4 align-text-bottom text-primary" /> 预期行为
- `<output>`：替换本轮用户输入。
- `<query>`：进入检索流程，提升召回命中率。

## 4. 3 条验证用例（建议立即执行）

1. 连续发送若干消息，确认达到间隔后出现总结任务。
2. 输入“我们之前约定的那件事”，检查召回日志是否返回相关历史。
3. 手动修改一条摘要并 Re-embed，再搜索新关键词验证检索更新。

## 5. 常见误区（新版）

- **误区 1**：只配对话模型，不配向量模型。
  - 结果：能生成总结，但文本检索和关联能力极弱或几乎无效。
- **误区 2**：楼层间隔设太小。
  - 结果：事件摘要碎片化严重，导致后续归纳合并的压力极大。
- **误区 3**：在记忆列表修改文本后，忘记点击重新向量化。
  - 结果：你看到的新文本与底层的数字向量索引不一致，导致搜索机制“像坏了一样”。
