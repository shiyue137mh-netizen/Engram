## 🐛 P0: 链路断了喵！这些逻辑会引发大问题！(ﾟДﾟ;)
| 危险度 | 案发地点（文件:行号） | 小码的深度剖析（发生了什么） | 修复思路喵 |
| :---: | :--- | :--- | :--- |
| 🔴 极度危险 | `src/data/sync/SyncService.ts` (Line 144) | **[主线程核爆级编码]**: 主人在这里用 `btoa(unescape(encodeURIComponent(jsonString)))` 把整个 IndexedDB 的聊天记录（包含海量事件和高维向量 Embedding）强行转成 Base64！当 JSON 字符串超过几MB时，`encodeURIComponent` 会在内存里创建体积放大三倍的巨型字符串，而这整个过程都是完全同步阻塞主线程的喵！不仅会让 UI 卡死（这就是为什么用户觉得“每次打完字过 3 秒页面就卡一下”），稍微大一点的数据还会直接报出 V8 的 `Maximum call stack size exceeded` 让插件当场去世！ | 请立刻扔掉这个古老的转换大法喵！如果酒馆 API 必须吃 Base64，建议切到 Web Worker 里去用 `FileReader` 配合 `Blob` 进行异步 Base64 编码，坚决不能阻塞主 UI 线程哦~ |

## ⚡ P1: 渲染有点浪费喵~ (´・ω・`)
| 紧急度 | 案发地点（文件:行号） | 哪里发生了无意义的重渲染/计算 | 优化方向喵 |
| :---: | :--- | :--- | :--- |
| 🟠 状态不同步 | `src/data/sync/SyncService.ts` (Line 32) | **[太被动的接收机制]**: 现在的同步下载 (`autoSyncDownload`) 仅仅在插件首次加载和触发 `CHAT_CHANGED`（切换聊天）时才会执行喵。如果主人在手机上打开同一个聊天，电脑端即使收到了服务器的更新文件，因为没有触发聊天切换，它就会在那一直装死不动！必须手动刷新或者切出去再切回来，难怪大家抱怨“不灵敏”呢。 | 在 `SyncService` 里监听 `visibilitychange` (当标签页重新获得焦点时)，或者干脆加上一个 15-30 秒的短轮询机制去 `getRemoteStatus` 检查远端时间戳喵~ |
